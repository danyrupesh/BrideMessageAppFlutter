# Search Module Upgrade Implementation - Architecture & Patterns

## Project Context
- Flutter app (BrideMessageApp) with Riverpod state management
- Two parallel readers: Bible (ReaderScreen) and Sermon (SermonReaderScreen)
- Search results from FTS database connect to these readers

## Key Architectural Patterns

### 1. Cross-Screen Data Transfer via Model
**Pattern:** Pass data between screens using provider model objects
- **Example:** ReaderTab model extends to carry `initialSearchQuery`
- **Advantage:** Type-safe, serializable, persists across sessions
- **File:** `lib/features/reader/models/reader_tab.dart`

### 2. Post-Frame Callback for State Initialization
**Pattern:** Use `WidgetsBinding.instance.addPostFrameCallback()` for side effects after build
- Prevents side-effects during render phase
- Ensures widgets are mounted before setState
- Used for: search activation, scroll positioning, match computation

```dart
if (tab.initialSearchQuery != null && !_isSearching) {
  WidgetsBinding.instance.addPostFrameCallback((_) {
    if (!mounted) return;
    setState(() { ... }); // Side effects here
  });
}
```

### 3. Provider-Specific Navigation
**Pattern:** Different content types use different providers
- Bible content → `readerProvider` → `/reader` route
- Sermon content → `sermonFlowProvider` → `/sermon-reader` route
- **Bug Fixed:** Sermon results were incorrectly using `readerProvider`

### 4. Nullable Optional Fields for Features
**Pattern:** Add nullable fields to models for optional features
- Non-breaking: existing code ignores null values
- Serializable: JSON methods handle null gracefully
- Example: `initialSearchQuery`, `verse` (for scrolling)

## State Management Insights

### Search Provider (Global)
- Manages search query, filters, results
- Used by SearchScreen to display results
- State: `query`, `bibleResults`, `sermonResults`, `isLoading`

### Reader Provider (Bible-Specific)
- Manages Bible reading tabs
- Methods: `openTab()`, `closeTab()`, `switchTab()`, `replaceCurrentTab()`
- Persists via `ReadingStateRepository`

### Sermon Flow Provider (Sermon-Specific)
- Manages sermon + Bible reference tabs
- Methods: `openSermon()`, `addSermonTab()`, `addBibleTab()`
- Difference: sermon is always tab[0], cannot be closed

## Implementation Learnings

### Session Persistence
- ReaderTab serialized via `readerTabToJson()` and `readerTabFromJson()`
- Stored in `ReadingFlowPayloadV1`
- Database: `ReadingStateRepository`
- **Important:** Always add new fields to JSON methods

### Search Logic Reuse
- In-page search is a self-contained feature in both readers
- Private state: `_isSearching`, `_searchController`, `_matchVerseIndices`
- Methods: `_computeMatches()`, `_navigateToMatch()`, `_buildSearchAppBar()`
- **Pattern:** Auto-activate just calls existing methods with pre-filled query

### UI Highlighting
- Two highlight styles: regular matches (blue) + current match (amber)
- Built via `_buildHighlightedSpans()` using regex and TextSpan
- Current match index tracked separately from total matches

## Critical Implementation Details

### What NOT to Do
1. ❌ Don't activate search during build (use post-frame callback)
2. ❌ Don't forget to check `if (!mounted)` before setState
3. ❌ Don't forget to add new fields to JSON serialization
4. ❌ Don't use wrong provider for content type
5. ❌ Don't leave unused imports after refactoring

### What TO Do
1. ✅ Use post-frame callbacks for state initialization
2. ✅ Always check mounted before setState in callbacks
3. ✅ Update both toJson() and fromJson() for model changes
4. ✅ Use type-specific providers and routes
5. ✅ Keep nullable fields nullable for backward compatibility
